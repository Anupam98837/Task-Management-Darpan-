Assignee Mobile App Implementation Handoff

This document is the implementation brief for building the assignee-side Flutter app against the current Laravel backend in this project. It is written to be reusable as input for another AI, a Flutter developer, or an app-builder workflow.

Derived from current codebase on 2026-05-04

1. Recommended Input Format

Use this HTML file as the source brief for the Flutter app. It is a better machine-readable handoff than a Word file because it keeps structure, API details, screen flow, permissions, and sample JSON together in one place.

If needed later, this same file can be converted into DOCX or PDF, but for AI-assisted app generation the HTML version is the best base document.

2. Product Goal

Build an Assignee Portal mobile app where a logged-in assignee can:

  • Log in securely with assignee credentials.
  • See a personal dashboard with KPI cards and charts.
  • View only jobs assigned to them.
  • Open a job and see summary details, assignees, chat messages, and expenses.
  • View and manage only their own job expenses where allowed by backend rules.
  • Create expense claim requests from their own expenses.
  • Read and manage notifications.
  • Optionally register FCM tokens for push notifications.

3. Important Backend Truths

Authentication Assignee auth uses custom Bearer tokens returned by /api/assignedpeople/login.
Role Validation After login, always verify role with /api/auth/context or /api/assignedpeople/me.
Job Scope Assignees must only see jobs where they are actively assigned.
Expense Scope Assignees only see expenses created by themselves inside a job.

Known UI/permission mismatch in current web code

The shared web job screen currently marks assignees as mostly read-only in some UI branches, but the backend still allows assignees to do several actions such as:

  • Change job status.
  • Post job chat messages.
  • Add expenses to a job they can access.
  • Create expense claims from their own expenses.

Recommendation for Flutter: follow backend capability, not the older shared Blade UI restrictions.

4. Authentication Flow

Login

Method Endpoint Body Success Output
POST /api/assignedpeople/login { "identifier": "email-or-name", "password": "..." } { status, message, access_token, token_type, tokenable_type }

Logout

Method Endpoint Auth Notes
POST /api/assignedpeople/logout Bearer token Invalidate token, then clear local auth storage.

Profile and role confirmation

  • GET /api/assignedpeople/me returns minimal assignee data: id, name, email.
  • GET /api/auth/context returns id, role, tokenable_type, name, email.
{
  "status": "success",
  "data": {
    "id": 12,
    "role": "assignee",
    "tokenable_type": "assignee",
    "name": "Rahul Sen",
    "email": "rahul@example.com"
  }
}

Flutter storage recommendation

  • Store access token in flutter_secure_storage.
  • Store only lightweight cache in memory or local persistence.
  • On app launch: read token, call /api/auth/context, confirm role is assignee, otherwise force logout.

7. My Jobs Screen UX Spec

Screen purpose

Show only jobs assigned to the logged-in assignee, with fast filters and a drill-down into job detail.

Recommended mobile structure

  • Top search bar.
  • Filter chips or modal filter sheet: Type, Priority, Status.
  • List as cards on mobile rather than a wide table.
  • Each job card should show: title, client_name, type, priority, status, deadline, assigned_at.
  • Tap card to open Job Detail screen.

Endpoints

Method Endpoint Purpose
GET /api/assignedpeople/my-jobs Main assignee list.
GET /api/job-details Fallback only if the primary endpoint fails.
GET /api/job-details/enums Populate filter options.

9. Conversation / Chat Spec

Use case

Assignee can participate in job-specific conversation threads and send attachments.

APIs

Method Endpoint Purpose
GET /api/job-details/{job}/messages?page=1&per_page=20 Paginated message list.
POST /api/job-details/{job}/messages Send message with optional attachments.
POST /api/job-details/{job}/messages/upload Optional rich-text inline image upload.
GET /api/job-details/messages/{messageId}/can-edit Check whether message can be edited.
PATCH /api/job-details/messages/{messageId} Edit a message.

Message create request

Use multipart form data.

message_html = "<p>Hello team</p>"
attachments[] = file1
attachments[] = file2

Message item example

{
  "id": 441,
  "job_id": 91,
  "sender_type": "assignee",
  "sender_id": 12,
  "sender_role": "assignee",
  "sender_name": "Rahul Sen",
  "sender_email": "rahul@example.com",
  "message_html": "<p>Please review this draft.</p>",
  "message_text": "Please review this draft.",
  "has_attachments": true,
  "attachments_count": 1,
  "attachments_json": [
    {
      "kind": "file",
      "original_name": "draft.docx",
      "mime": "application/vnd.openxmlformats-officedocument.wordprocessingml.document",
      "absolute_url": "https://...",
      "relative_url": "/f/jobMessages/..."
    }
  ],
  "created_at": "2026-05-04 13:40:00",
  "updated_at": "2026-05-04 13:40:00"
}

Implementation notes

  • Show message bubble alignment by sender role / sender id.
  • Support attachments preview list.
  • Allow editing only if backend says it is allowed.
  • If Flutter has trouble sending multipart with HTTP PATCH, send POST multipart with _method=PATCH as fallback, because the current web UI does this.

11. Expense Claim / Reimbursement Flow

Goal

The assignee can request reimbursement for an expense they created themselves.

Flow

  1. Assignee picks a job.
  2. App loads expenses for that job.
  3. App shows only expenses created by current assignee.
  4. Assignee taps Claim on one expense.
  5. App posts claim request.
  6. Claim history screen shows submitted claim statuses.

APIs

Method Endpoint Purpose
POST /api/job-expense-claims/claim Create claim request for one expense.
GET /api/job-expense-claims/my Claim history for logged-in assignee.

Create claim request body

{
  "expense_id": 51,
  "job_id": 91,
  "message": "Please reimburse this travel expense."
}

  • Mark many as read.

16. Open Decisions for Mobile

  • Whether assignees should be allowed to change job status in the mobile app. Backend allows it.
  • Whether assignees should be allowed to add expenses from inside job detail or only from a dedicated expense screen. Backend allows add/edit within limits.
  • Whether chat editing UI should be shown by default or only after can-edit confirms availability.
  • Whether exports should open in external file viewer or just download/share from app.

Build a Flutter mobile app for the "Assignee Portal" using the Laravel APIs documented in this file.

Primary requirements:
1. Login with /api/assignedpeople/login using identifier + password.
2. Store Bearer token securely.
3. Confirm role with /api/auth/context and only allow role=assignee.
4. Dashboard must show:
   - Assigned Jobs
   - Due Today
   - Overdue

   - Status pie chart using /api/assignedpeople/status-stats

   - list messages
   - send message
   - file attachments
   - edit message if backend allows
8. Expenses must support:
   - list assignee-owned expenses in a job
   - create expense
   - edit expense within backend rules
9. Expense Claims must support:
   - submit claim for one expense
   - list my claims
10. Notifications must support:
   - unread badge
   - inbox
